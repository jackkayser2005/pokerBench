About PokerBench

Deterministic evaluation for heads-up LLM poker agents

PokerBench is a laboratory for measuring autonomous No-Limit Hold'em decision making. Mirrored-seed duels, strict legality checks, and exhaustive telemetry keep every experiment reproducible. Elo and Glicko-2 pipelines expose skill progression, while a Monte-Carlo EV judge audits whether models understand the cards they are dealt.

Mirrored Seeds
Seat swaps neutralize position bias; identical shuffles isolate strategic strength.
Ratings Science
Career Elo with a tanh chip model plus τ = 0.5 Glicko-2 uncertainty tracking.
Monte-Carlo Judge
Post-match rollouts estimate EV accuracy so prompt regressions surface immediately.

Scientific pillars

Every capability is designed to be audited, replayed, and repeated without hidden randomness.

  • Mirrored pairs. Each seed plays both seats (SB/BB) with the same deck order so chip deltas reflect policy, not luck.
  • Deterministic orchestration. Seeds drive pairings, deck shuffles, and evaluation windows, enabling byte-for-byte reruns.
  • Strict legality enforcement. The engine validates to_call, minimum raises, and all-in bounds before any action is accepted.
  • Telemetry-first logging. Structured action logs, stack snapshots, and board states back the Replay, History, and analytics views.
  • Provider agnostic. Any OpenAI-compatible endpoint (OpenAI, Azure, OpenRouter, Together, Groq, …) can be slotted in.

System architecture at a glance

A lean Go stack keeps the benchmark predictable and inspectable.

engine/

Heads-up logic

Card utilities and betting flow for deterministic heads-up Hold'em, paired with agent/ for structured observations.

main.go & router.go

Scheduler + API

Coordinates mirrored matches, exposes JSON APIs, and serves the static UI from web/.

judge/ & stats.go

Ratings lab

Monte-Carlo EV auditing plus Elo & Glicko-2 updates that translate chip margins into interpretable skill ratings.

store/

Postgres schema

Normalized tables (matches, action_logs, bot_ratings, …) persist every hand, rating, and telemetry stream.

Experimental modes

Server mode

Always-on lab

Run the web UI, JSON APIs, scheduler, and database together. Ideal for live rating updates and public dashboards.

Duel mode

Single experiment

Launch a mirrored duel from the CLI to interrogate a prompt or compare two agents under controlled blinds and stacks.

Duel matrix

Bulk evaluation

Schedule multiple model matchups across a grid of seeds to map the meta-game and stress test new strategies.

Data fidelity & observability

Everything you see on the Leaderboard, Matrix, Elo, and History pages is backed by first-class telemetry.

Action stream
action_logs

Per-action snapshots containing actor, stacks, bounds, board cards, and the legal window.

Career views
v_bot_career

Aggregates matches, net BB, hands played, judge accuracy, and update timestamps.

Rating history
rating_history

Stores Elo and Glicko-2 checkpoints so uncertainty can be tracked over time.

Career Elo uses E_A = 1 / (1 + 10^{(R_B - R_A)/400}) with a tanh-scaled chip score. Glicko-2 follows the 173.7178 scaling with volatility solving per rating period.

Agent integration workflow

  1. Emit the contract. Implement the structured JSON schema (fold, check/call, raise with a legal size) and avoid prose.
  2. Validate legality. Run actions through the engine’s to_call and raise bounds locally before scheduling matches.
  3. Stress mirrored seeds. Use paired seats to confirm ranges hold up, then replay hands and inspect judge EV for regressions.

Operational playbook

Docker compose
Option A

Copy compose.env, drop secrets in secrets/, and run docker compose up --build to launch Postgres, migrations, duel runner, and the web server.

Local Go
Option B

Use Go 1.21+, export an OpenAI-compatible API key and Postgres URL, build ./ai-thunderdome, and start the server with AUTO_MIGRATE=1.

Governance & openness

  • Reproducibility safeguards. Illegal or malformed agent output is rejected and logged; sampling tags surface provisional data.
  • Security awareness. Secrets stay in secrets/, and the stack is ship-ready via Docker or local binaries.
  • Open science. Released under AGPL-3.0-or-later with transparent schemas, logs, and UI so results can be audited end-to-end.
